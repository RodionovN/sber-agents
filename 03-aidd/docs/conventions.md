# Правила разработки кода

## Общие принципы

- **KISS** — максимально простое решение. Избегай сложных абстракций и паттернов.
- **YAGNI** — не добавляй функциональность "на будущее". Реализуй только то, что нужно сейчас.
- **Простота превыше всего** — если есть выбор между простым и "правильным" решением, выбирай простое.

## Структура кода

- Один файл — одна ответственность.
- Функции должны быть короткими и делать одну вещь.
- Избегай глубокой вложенности (максимум 2-3 уровня).
- Не создавай классы без необходимости — используй функции, если достаточно.

## Именование

- Имена переменных и функций должны быть понятными и отражать назначение.
- Используй snake_case для переменных и функций.
- Используй UPPER_CASE для констант.
- Избегай сокращений, если они не очевидны.

## Обработка ошибок

- Обрабатывай только те ошибки, которые можешь обработать осмысленно.
- Для остальных — логируй и пробрасывай выше или завершай выполнение.
- Не создавай сложные иерархии исключений — используй стандартные.

## Импорты и зависимости

- Импортируй только то, что используешь.
- Группируй импорты: стандартная библиотека, сторонние, локальные.
- Избегай циклических зависимостей.

## Комментарии

- Код должен быть самодокументируемым.
- Комментируй только "почему", а не "что".
- Удаляй закомментированный код.

## Тестирование

- Пиши код так, чтобы его было легко тестировать.
- Но не создавай тесты "на будущее" — только для критичной функциональности.
